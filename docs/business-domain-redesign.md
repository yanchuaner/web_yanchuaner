# 业务域重构方案

最后更新：2026-07-17

## 1. 产品定位

燕中校友数字母港是以身份可信为基础的私域校友社区。公开站点负责解释平台、建立信任和引导注册；认证后的成员空间负责校友连接、内容沉淀与活动协作；后台负责审核、运营和审计。

本轮不把平台扩展成通用学校官网或社交网络。新增能力必须归入以下六个业务域，并有清晰的用户入口、管理入口、状态反馈和退出路径。

| 业务域 | 用户侧 | 管理侧 |
| --- | --- | --- |
| 访问与认证 | 注册、邮箱验证、内部口令、登录、密码恢复 | 注册策略、人工审核、身份审核 |
| 成员与名录 | 个人资料、校友地图、资料纠错、纪念卡 | 用户、名录、纠错审核 |
| 社区内容 | 故事、新闻、记忆、成就、学生与教师资源 | 投稿审核、内容发布、页面内容 |
| 活动协作 | 活动浏览、报名、报名状态 | 活动、名额、报名名单 |
| 站点内容 | 首页、关于、生态、隐私、联系 | 页面内容配置 |
| 系统治理 | 账户安全与隐私设置 | 管理员权限、会话、审计日志 |

## 2. 注册与认证策略

### 2.1 新流程

注册只保留一份表单和两条可解释路径：

1. 用户创建账号并验证邮箱。
2. 用户可填写管理员线下提供的内部口令。
3. 口令有效：账号预先通过校友认证，邮箱验证后立即可用。
4. 未填写或口令无效：账号正常创建，邮箱验证后进入人工审核。

内部口令不能绕过邮箱验证，不能在任何 API 中以明文返回。数据库只保存 bcrypt 哈希；管理员可以启用、停用和轮换口令，并可配置不包含秘密的提示语。所有策略变更写入审计日志。

### 2.2 名单的职责

校友名单不再作为注册门槛。它继续承担校友地图、搜索、纪念卡和资料治理的数据来源。通过内部口令或人工审核成为校友后，再将已确认资料同步到名单。

### 2.3 旧资料认领退役

注册不再创建旧资料认领申请。`UserClaimRequest` 模型、后台页面和 API 已通过独立迁移完整删除；校友历史资料继续由名单管理与信息修正流程维护。

## 3. 后台信息架构

后台导航收束为：

- 总览
- 成员与认证：注册策略、用户审核、身份认证、信息修正、校友名单
- 社区内容：故事审核、故事、新闻、活动、成就、记忆
- 站点与治理：页面内容，后续加入审计日志

旧帖子和独立教师频道不再占用一级导航。旧帖子暂时作为历史数据保留；教师内容统一由 `ContentSection(page=teachers)` 管理。

## 4. 状态与可观测性

用户认证来源使用 `verificationMethod` 记录：

- `INTERNAL_CODE`：内部口令
- `ADMIN_REVIEW`：网页注册人工审核
- `IDENTITY_REVIEW`：身份认证申请审核
- `LEGACY`：迁移前已认证账号或无法确认来源

后台用户列表应同时展示邮箱状态、认证状态和认证来源，避免管理员根据 `role` 猜测账号处于哪个环节。

## 5. 分阶段收束

### 第一阶段：本轮

- 可配置内部口令及安全存储
- 注册双路径和双语反馈
- 认证来源记录
- 旧资料认领退役
- 后台导航、总览和待办收束

### 第二阶段

- 网页活动增加“我的报名”、报名状态和取消报名
- 成就投稿要么形成审核闭环，要么永久移除投稿入口
- `Post` 历史数据迁移到 `Story` 后删除旧模型

### 第三阶段

- 删除归档的 `TeacherSection`
- 建立可检索、可导出的审计日志页面
- 统一网站与小程序的身份类型和权限矩阵

## 6. 验收标准

- 未配置口令时，注册行为与当前人工审核流程兼容。
- 有效口令不能绕过邮箱验证，邮箱验证完成后账号立即可用。
- 无效口令不泄露正确口令信息，不阻止注册，只进入人工审核。
- 管理员页面永不返回口令明文或哈希。
- 普通访客不能读取或修改后台注册策略。
- 注册页、验证页、登录状态页和后台设置均支持中英文与亮暗主题。
- 旧资料认领不再出现在注册表单、后台导航和待办统计中。
