2026 燕中生态暑期预览 / Yanchuan Ecosystem Summer Preview
更新日期：2026-07-21

阶段 0 状态：`DEVELOPMENT_READY`。燕中校友汇具备公益运营资格，工程、版权和发布边界已经登记；Open WebUI 无品牌授权不阻止自主研发，但在自主入口完成前不得把 Open WebUI 包装成燕中原创产品或据此扩大公开稳定服务。

阶段 1 状态：`IMPLEMENTED_PARTIAL_E2E`。YanCore Subject Grant、主站 S256 PKCE、受控主体交换、15 分钟逐用户应用 Key 和自主 AI Web SSE 对话链路已经实现；隔离 Docker 环境已验证 LiteLLM 三模型路由、API 受限 Key、真实额度结算、不可变流水、用量日志与 request ID。下一阻断项是 Linux/HTTPS staging 中的主站 OIDC 端到端、真实 OpenAI/DeepSeek、失败退款和跨渠道验收。Open WebUI 与 New API 只保留为可替换的过渡客户端/兼容网关。

阶段 2 状态：`IMPLEMENTED_LOCAL_ACCEPTANCE_PENDING`。活动/兑换码/权益来源分账和同步结算已经实现；阶段 2D 已提供自主虚拟 Key 策略控制台，并在同一事务中维护预算、有效期、模型、来源 IP、OpenAI/DeepSeek 供应商、RPM、TPM、并发、状态和不可变修订。旧 Token 写入口对策略 Key 返回 409，避免绕过策略审计；异步、媒体与任务协议在具备持久化预扣/结算前 fail-closed。SQLite、PostgreSQL 16.14、MySQL 8.0、Redis 7、前端构建和生产依赖审计已通过本地验证；下一步是历史 Key 回填审查和真实 OpenAI/DeepSeek/OIDC 验收。

本轮隔离验收使用全新 PostgreSQL 卷和本机 OpenAI 兼容测试上游，不包含真实供应商密钥或生产数据；Windows Docker Desktop 的宿主回环代理和 Docker Hub Node 镜像拉取仍阻断自主 AI Web 容器级验收，不能将测试上游结果宣称为真实供应商验收。

阶段 0 决策真值分别维护在 `api_yanchuaner/docs/yanchuaner/phase-0-decision-record.md` 与 `ai_yanchuaner/docs/phase-0-gate.md`。`DEVELOPMENT_READY` 只表示自主研发可以继续，不代表外部授权、隐私、生产权限或公开稳定发布已经放行；门禁不能由单个仓库自行解除。

一、发布定位

这次更新不命名为简单的 v2 或 v3，而是燕中生态第一次完整的联合预览。
暑期结束目标是让“一主站、五子域、一个小程序、一个 Agent 产品”都有清晰入口、身份边界、可演示流程和公开文档。

暑期发布的是 Preview，不等于正式版。正式版计划在大二上学期持续一个学期，根据真实成员反馈、模型成本、安全审计、移动端体验和运维数据不断打磨后发布。

二、一主站与五子域

1. yanchuaner.cn / web_yanchuaner
   状态：生态核心已进入联合预览准备；OIDC 已支持 S256 PKCE、state、nonce、RS256 ID Token 与一次性授权码。
   职责：唯一身份源、生态首页、成员认证、校友内容、活动、个人中心与运营后台。
   边界：不保存模型厂商密钥，不承担模型计费，不复制子站业务数据库。

2. ai.yanchuaner.cn / ai_yanchuaner
   状态：Open WebUI/LiteLLM PoC 保留；自主 `apps/ai-web` 已完成主站 OIDC、加密 HttpOnly 会话、YanCore 用户主体、模型白名单、会话预算、SSE BFF 和最小对话界面。隔离环境已验证 API 逐用户计费下游，真实主站 OIDC、真实供应商、退款和容器级运行仍待 staging 验收。
   职责：为认证成员提供网页 AI 工作台，承载对话、内容辅助和后续轻量工具。
   边界：不维护第二套公开注册，不作为用户余额真值，不公开 LiteLLM 管理端；Open WebUI 共享 Key 只记独立服务账户，自主 AI Web 短期 Key 才允许进入个人公益额度链路，两者不得混账。

3. api.yanchuaner.cn / api_yanchuaner
   状态：主站 OIDC 与过渡控制面链路已完成；B2 的哈希虚拟 Key、不可变额度流水和钱包预扣/结算/退款已完成验证；阶段 1 已实现 YanCore 主体凭证、固定 UserInfo 复验、并发会话指针与逐用户短期应用 Key；阶段 2 已实现独立活动、哈希兑换码、用户权益、领取幂等和 `campaign` 来源流水；阶段 2B 已接入可开关的同步活动权益预扣、结算与退款；阶段 2D 已实现自主虚拟 Key 策略控制台、策略与兼容 Token 投影的原子更新、不可变修订快照和旧写入口防绕过，并在 PostgreSQL 16.14、MySQL 8.0、Redis 7 与本地容器验证。功能默认关闭，仍待历史 Key 回填审查、异步持久化结算、真实 OpenAI/DeepSeek、主站 OIDC 和 staging 验收。
   职责：模型 API 控制面，管理用户密钥、模型权限、公益额度、限流、用量和审计。
   规则：关闭本地密码登录；首期新成员公益额度为 1 USD / 7 CNY；新虚拟 Key 只显示一次且仅保存哈希；管理员调额必须填写原因并追加不可变流水。

四、唯一身份与模型调用链

用户 API 链路：
认证成员
   -> yanchuaner.cn 完成邮箱和在校生 / 校友 / 教师身份审核
   -> OIDC 登录燕中 API，领取公益权益并创建按应用/场景命名的个人虚拟 Key
   -> 燕中 API 执行来源 IP、模型、OpenAI/DeepSeek 供应商、有限预算、有效期、每 Key RPM/TPM/并发、额度预扣/结算/退款和审计
   -> LiteLLM 执行路由、重试和上游成本核对
   -> 首期获授权的 OpenAI Platform API Project 或 DeepSeek 官方 API

AI 工作台链路：
认证成员
   -> S256 PKCE + state + nonce 登录自主燕中 AI Web
   -> AI BFF 用独立服务凭据把短期主站令牌提交给燕中 API
   -> 燕中 API 通过固定 UserInfo 地址复验并映射已绑定用户
   -> 签发最长 15 分钟的 YanCore Subject Grant
   -> 一次性返回同期限、有限预算、精确模型白名单的应用 Key，数据库仅保存哈希
   -> AI BFF 同源校验后以 SSE 调用燕中 API /v1
   -> 燕中 API 执行模型/供应商/每 Key 速率策略、公益额度预扣/结算/退款与用量日志
   -> LiteLLM 路由到获授权上游并提供成本核对

Open WebUI 过渡链路继续使用燕中 API 受限服务 Key 并暂记独立服务账户，不继承自主 AI Web 的个人主体，也不得静默扣减个人公益额度。

主站是身份真值；燕中 API 的不可变流水是权益与消费账本真值，users.quota 仅作兼容投影；公益、活动权益和未来 BYOK 使用不同 funding source，不能静默互相扣费；活动权益消费和每 Key 策略开关默认关闭，回填无法识别供应商的历史 Key 必须保持禁用，Redis 已配置但故障时限流必须 fail-closed；匹配活动额度不足时请求拒绝，范围不匹配时钱包只可扣公益/兼容旧余额；策略 Key 当前仅允许已接入同步预扣/结算的 OpenAI Chat/Responses 协议，异步、媒体和任务请求在持久化来源标识与结算完成前直接拒绝；LiteLLM 费用只用于上游成本核对。各项目不共享登录 Cookie，不复制主站用户表，不把真实密钥提交到 Git。

五、暑期预览统一完成标准

- 每个公开入口都有清晰的中英定位、亮暗主题策略和移动端可用布局；纯中文内容允许渐进翻译，但界面导航必须统一。
- 主站、AI、API、Lab、小程序和 YCZX Code 均有可重复的本地运行或演示步骤。
- ai.* 与 api.* 只允许主站认证成员进入；论坛与生日服务未完成隐私和治理前只做说明或受控演示。
- 公益额度有限且可解释，部署重启不得自动补满余额；管理员发放和高风险操作必须可审计。
- 每个仓库 README 说明当前状态、暑期预览目标、正式版门槛、边界和与其他项目的关系。
- 核心链路具备自动化检查、备份恢复说明、故障回滚路径和至少一次真实端到端验收。

阶段 0 退出前的额外硬门槛：

- 运营主体、贡献协议和自主代码许可证已有批准记录；
- Open WebUI 品牌处理和 OpenAI/DeepSeek 代理访问授权有可复核证据；
- 生产域名、证书、Secret、备份和管理员权限可由团队交接；
- 首批用户范围、数据保留/删除、支持渠道和停服机制已公开；
- 共享服务 Key 不能按个人归因时，只能独立记入 AI 服务账户，不得扣个人公益额度。
- 每个仓库保留 LICENSE、NOTICE、第三方清单、来源矩阵和安全披露流程；界面换名不改变上游源码版权。

上述门槛约束公开稳定发布，不阻止在隔离开发/测试环境继续实现 YanCore、自主 AI Web、YCZX Code 和兼容性测试。

六、大二上学期正式版方向

- 根据一个学期的灰度数据收束功能，不以版本号替代质量判断。
- 完善账号停用同步、模型故障切换、预算告警、长期审计、隐私授权和运营规则。
- 评估合规 BYOK、凭据保险库、论坛治理、生日自动化与更多 Agent 产品。
- 只有通过真实用户流程、移动端、成本、安全、备份和故障演练后，才将对应产品标记为正式版。

七、仓库与职责登记规则

新增项目或子域前，必须在本文件登记：产品对象、公开入口、仓库、身份来源、数据所有者、调用方向、当前阶段和退出机制。
同一能力只允许一个真值来源；已有主站身份、燕中 API 额度或 LiteLLM 路由能力时，子项目必须复用，不得重复建设。

八、版权与替换边界

- api_yanchuaner 当前是 New API 的 AGPLv3 授权修改版；必须保留上游 LICENSE、NOTICE、版权头、界面署名与链接。自主额度流水和哈希密钥模块有独立设计与测试，但不把相邻上游代码改称原创。
- ai_yanchuaner 的 `apps/ai-web`、YanCore 客户端、加密会话、SSE 代理与燕中对话界面为独立需求和实现；Next.js、React、openid-client 与 Lucide 是明确登记的第三方依赖。LiteLLM 继续作为可替换路由依赖，Open WebUI 作为过渡客户端且不复制其源码。
- 未取得 Open WebUI 书面或企业许可时，燕中品牌部署必须保持滚动 30 日直接用户不超过 50 人，并在法律声明中提供其许可与版权信息。
- 燕中自主内容的最终许可证须在运营主体和贡献政策确认后选择。普通贡献不会自动把版权转让给组织，第三方代码版权始终归第三方权利人。
- 每完成一个自主模块，必须同步更新来源矩阵、替换进度、数据迁移、兼容测试与回滚说明；禁止通过改名、格式化或 AI 改写将衍生代码标记为原创。
