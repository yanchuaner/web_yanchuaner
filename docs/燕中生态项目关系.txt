2026 燕中生态暑期预览 / Yanchuan Ecosystem Summer Preview
更新日期：2026-07-17

一、发布定位

这次更新不命名为简单的 v2 或 v3，而是燕中生态第一次完整的联合预览。
暑期结束目标是让“一主站、五子域、一个小程序、一个 Agent 产品”都有清晰入口、身份边界、可演示流程和公开文档。

暑期发布的是 Preview，不等于正式版。正式版计划在大二上学期持续一个学期，根据真实成员反馈、模型成本、安全审计、移动端体验和运维数据不断打磨后发布。

二、一主站与五子域

1. yanchuaner.cn / web_yanchuaner
   状态：生态核心已进入联合预览准备。
   职责：唯一身份源、生态首页、成员认证、校友内容、活动、个人中心与运营后台。
   边界：不保存模型厂商密钥，不承担模型计费，不复制子站业务数据库。

2. ai.yanchuaner.cn / ai_yanchuaner
   状态：主站 OIDC、燕中 API 服务 Key、Open WebUI 与 LiteLLM 已打通。
   职责：为认证成员提供网页 AI 工作台，承载对话、内容辅助和后续轻量工具。
   边界：不维护第二套公开注册，不作为用户余额真值，不公开 LiteLLM 管理端。

3. api.yanchuaner.cn / api_yanchuaner
   状态：统一身份、API Key、双币额度、逐请求账单和管理员审计已完成预览链路。
   职责：模型 API 控制面，管理用户密钥、模型权限、公益额度、限流、用量和审计。
   规则：关闭本地密码登录；首期新成员公益额度为 1 USD / 7 CNY；管理员调额必须填写原因。

4. lab.yanchuaner.cn / lab_yanchuaner
   状态：可运行 Demo、设计系统、教程框架、项目展示和发布检查已建立。
   职责：科创教程、学习路径、真实项目展示、仓库链接与第一次开源贡献引导。
   暑期目标：完成开源协作、App/小程序、Agent 三类入口，以中文主内容为准，英文渐进补齐。

5. forum.yanchuaner.cn（拟用前缀）/ 规划中
   状态：暑期产品预览目标，尚未建设独立仓库和正式社区。
   职责：文史政哲、校园议题和校友观察的轻量讨论空间，比主站投稿更开放。
   暑期目标：至少完成公开说明、社区规则、内容分类和可演示发帖流程；未完成治理与审核前不开放匿名发帖。

6. birthday.yanchuaner.cn / 规划中
   状态：暑期产品预览目标，尚未建设独立仓库和自动发送服务。
   职责：在成员明确授权和隐私保护前提下提供生日祝福、纪念页面与校友关怀。
   暑期目标：至少完成公开说明、授权流程、手动演示页面和退出机制；不得从通讯录默认推断并公开生日。

三、跨端与 Agent 产品

1. 燕中校友汇微信小程序 / mp_yanchuaner
   定位：主站的移动服务端，不是独立社区。
   当前能力：微信登录、身份申请、资料维护、动态、活动报名、隐私说明与账号注销。
   暑期目标：完成真实设备、弱网、会话恢复、隐私授权与活动闭环验收。

2. YCZX Code / yczx_code
   定位：燕中生态首个轻量 CLI Coding Agent。
   暑期目标：发布可安装 Preview，具备 Agent Loop、文件和命令工具、权限确认、上下文管理、会话日志、测试与代表性 Demo。
   模型入口：优先使用 api.yanchuaner.cn/v1，不在 Agent 中重复建设模型聚合、额度和账单系统。

四、唯一身份与模型调用链

认证成员
   -> yanchuaner.cn 完成邮箱和在校生 / 校友 / 教师身份审核
   -> OAuth / OIDC 登录燕中 API 与燕中 AI
   -> 燕中 API 签发用户或受限服务 Key，执行额度、模型权限、限流和审计
   -> LiteLLM 执行模型路由、重试和上游成本核对
   -> 获授权的 OpenAI Platform API Project、DeepSeek 官方 API 等上游

主站是身份真值；燕中 API 是用户额度和消费账本真值；LiteLLM 费用只用于上游成本核对。各项目不共享登录 Cookie，不复制主站用户表，不把真实密钥提交到 Git。

五、暑期预览统一完成标准

- 每个公开入口都有清晰的中英定位、亮暗主题策略和移动端可用布局；纯中文内容允许渐进翻译，但界面导航必须统一。
- 主站、AI、API、Lab、小程序和 YCZX Code 均有可重复的本地运行或演示步骤。
- ai.* 与 api.* 只允许主站认证成员进入；论坛与生日服务未完成隐私和治理前只做说明或受控演示。
- 公益额度有限且可解释，部署重启不得自动补满余额；管理员发放和高风险操作必须可审计。
- 每个仓库 README 说明当前状态、暑期预览目标、正式版门槛、边界和与其他项目的关系。
- 核心链路具备自动化检查、备份恢复说明、故障回滚路径和至少一次真实端到端验收。

六、大二上学期正式版方向

- 根据一个学期的灰度数据收束功能，不以版本号替代质量判断。
- 完善账号停用同步、模型故障切换、预算告警、长期审计、隐私授权和运营规则。
- 评估合规 BYOK、凭据保险库、论坛治理、生日自动化与更多 Agent 产品。
- 只有通过真实用户流程、移动端、成本、安全、备份和故障演练后，才将对应产品标记为正式版。

七、仓库与职责登记规则

新增项目或子域前，必须在本文件登记：产品对象、公开入口、仓库、身份来源、数据所有者、调用方向、当前阶段和退出机制。
同一能力只允许一个真值来源；已有主站身份、燕中 API 额度或 LiteLLM 路由能力时，子项目必须复用，不得重复建设。
