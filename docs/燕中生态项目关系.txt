一、发布定位

这次更新不命名为简单的 v2 或 v3，而是燕中生态第一次完整的联合预览。
暑期结束目标是让“一主站、五子域、一个小程序、一个 Agent 产品”都有清晰入口、身份边界、可演示流程和公开文档。

暑期发布的是 Preview，不等于正式版。正式版计划在大二上学期持续一个学期，根据真实成员反馈、模型成本、安全审计、移动端体验和运维数据不断打磨后发布。

二、一主站与五子域

1. yanchuaner.cn / web_yanchuaner
   状态：生态核心已进入联合预览准备。
   职责：唯一身份源、生态首页、成员认证、校友内容、活动、个人中心与运营后台。
   边界：不保存模型厂商密钥，不承担模型计费，不复制子站业务数据库。

2. ai.yanchuaner.cn / ai_yanchuaner
   状态：主站 OIDC、燕中 API 服务 Key、Open WebUI 与 LiteLLM 已打通；共享服务 Key 尚不能证明逐用户调用归因。
   职责：为认证成员提供网页 AI 工作台，承载对话、内容辅助和后续轻量工具。
   边界：不维护第二套公开注册，不作为用户余额真值，不公开 LiteLLM 管理端；用户级归因完成前只记独立服务账户，不静默扣减个人公益额度。

3. api.yanchuaner.cn / api_yanchuaner
   状态：主站 OIDC 与过渡控制面链路已完成；B2 的哈希虚拟 Key、不可变额度流水和钱包预扣/结算/退款已完成代码、单元测试、PostgreSQL 16.14 迁移和完整镜像构建，仍待真实主站与模型上游端到端验收。
   职责：模型 API 控制面，管理用户密钥、模型权限、公益额度、限流、用量和审计。
   规则：关闭本地密码登录；首期新成员公益额度为 1 USD / 7 CNY；新虚拟 Key 只显示一次且仅保存哈希；管理员调额必须填写原因并追加不可变流水。

   定位：燕中生态首个轻量 CLI Coding Agent。

   模型入口：优先使用 api.yanchuaner.cn/v1，不在 Agent 中重复建设模型聚合、额度和账单系统。

四、唯一身份与模型调用链

用户 API 链路：
认证成员
   -> yanchuaner.cn 完成邮箱和在校生 / 校友 / 教师身份审核
   -> OIDC 登录燕中 API，领取公益权益并创建按应用/场景命名的个人虚拟 Key
   -> 燕中 API 执行模型权限、有限预算、额度预扣/结算/退款和审计
   -> LiteLLM 执行路由、重试和上游成本核对
   -> 首期获授权的 OpenAI Platform API Project 或 DeepSeek 官方 API

AI 工作台链路：
认证成员
   -> OIDC 登录燕中 AI / Open WebUI
   -> Open WebUI 使用燕中 API 受限服务 Key
   -> 燕中 API 暂记独立服务账户
   -> 完成可信用户身份透传或令牌交换后，才能按个人权益记账

主站是身份真值；燕中 API 的不可变流水是权益与消费账本真值，users.quota 仅作兼容投影；LiteLLM 费用只用于上游成本核对。各项目不共享登录 Cookie，不复制主站用户表，不把真实密钥提交到 Git。

五、暑期预览统一完成标准

- 每个公开入口都有清晰的中英定位、亮暗主题策略和移动端可用布局；纯中文内容允许渐进翻译，但界面导航必须统一。
- 主站、AI、API、Lab、小程序和 YCZX Code 均有可重复的本地运行或演示步骤。
- ai.* 与 api.* 只允许主站认证成员进入；论坛与生日服务未完成隐私和治理前只做说明或受控演示。
- 公益额度有限且可解释，部署重启不得自动补满余额；管理员发放和高风险操作必须可审计。
- 每个仓库 README 说明当前状态、暑期预览目标、正式版门槛、边界和与其他项目的关系。
- 核心链路具备自动化检查、备份恢复说明、故障回滚路径和至少一次真实端到端验收。

阶段 0 退出前的额外硬门槛：

- 运营主体、贡献协议和自主代码许可证已有批准记录；
- Open WebUI 品牌处理和 OpenAI/DeepSeek 代理访问授权有可复核证据；
- 生产域名、证书、Secret、备份和管理员权限可由团队交接；
- 首批用户范围、数据保留/删除、支持渠道和停服机制已公开；
- 共享服务 Key 不能按个人归因时，只能独立记入 AI 服务账户，不得扣个人公益额度。
- 每个仓库保留 LICENSE、NOTICE、第三方清单、来源矩阵和安全披露流程；界面换名不改变上游源码版权。

六、大二上学期正式版方向

- 根据一个学期的灰度数据收束功能，不以版本号替代质量判断。
- 完善账号停用同步、模型故障切换、预算告警、长期审计、隐私授权和运营规则。
- 评估合规 BYOK、凭据保险库、论坛治理、生日自动化与更多 Agent 产品。
- 只有通过真实用户流程、移动端、成本、安全、备份和故障演练后，才将对应产品标记为正式版。

七、仓库与职责登记规则

新增项目或子域前，必须在本文件登记：产品对象、公开入口、仓库、身份来源、数据所有者、调用方向、当前阶段和退出机制。
同一能力只允许一个真值来源；已有主站身份、燕中 API 额度或 LiteLLM 路由能力时，子项目必须复用，不得重复建设。

八、版权与替换边界

- api_yanchuaner 当前是 New API 的 AGPLv3 授权修改版；必须保留上游 LICENSE、NOTICE、版权头、界面署名与链接。自主额度流水和哈希密钥模块有独立设计与测试，但不把相邻上游代码改称原创。
- ai_yanchuaner 只编排 LiteLLM、Open WebUI 与 PostgreSQL 镜像，不复制其源码。LiteLLM 继续作为可替换路由依赖；Open WebUI 作为过渡客户端并计划逐步替换核心交互层。
- 未取得 Open WebUI 书面或企业许可时，燕中品牌部署必须保持滚动 30 日直接用户不超过 50 人，并在法律声明中提供其许可与版权信息。
- 燕中自主内容的最终许可证须在运营主体和贡献政策确认后选择。普通贡献不会自动把版权转让给组织，第三方代码版权始终归第三方权利人。
- 每完成一个自主模块，必须同步更新来源矩阵、替换进度、数据迁移、兼容测试与回滚说明；禁止通过改名、格式化或 AI 改写将衍生代码标记为原创。
